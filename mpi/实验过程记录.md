## 阶段 1：MPI IVF 基础实现

### 2026-06-03 初始化

**创建文件:**
- `mpi/mpi_utils.h` — MPI 通用工具（序列化、top-k merge、recall 计算）
- `mpi/main_ivf_mpi.cc` — IVF MPI 主程序

**设计决策:**

1. **按簇分配 inverted list**（而非按向量 ID 切分）
   - 理由: 保持每个 rank 上 SIMD 扫描的连续性，每个 rank 获得若干完整的簇
   - 按向量 ID 切分会破坏 cache locality，且需要在 rank 间传递大量 base 向量数据

2. **使用全局变量传递 IVF 索引**
   - `ivf_simd.h` 使用 `static` 全局变量 (`g_ivf_centroids`, `g_ivf_lists`, `g_ivf_nlist`, `g_ivf_base`)
   - 每个 MPI rank 设置自己的全局变量副本，避免跨进程共享
   - 优点: 不需要修改已有的 `ivf_search()` 函数签名

3. **top-k 归并方式**
   - 提供两种模式对比:
     - `gather`: MPI_Gather 收集所有 rank 的 local top-k 到 rank 0，串行 merge
     - `reduce`: 自定义 MPI_Op 实现树形规约（O(log P) 轮通信）
   - 使用环境变量 `IVF_MERGE_MODE` 切换

4. **时序测量**
   - 使用 `MPI_Wtime()` 替代 `gettimeofday()`，因为 MPI 提供了跨进程的时间基准
   - 在 rank 0 上测量端到端延迟（包含通信开销）

### 遇到的问题和解决方法

**问题 1: recall 接近 0 (0.00065)**
- 原因: `compute_recall()` 中 ground truth 索引错误。函数签名接受指向当前 query 的 gt 起始位置的指针，但内部使用了 `gt[j * gtd]`（应为 `gt[j]`，因为指针已指向正确位置）。同时调用处使用了 `test_gt + qi` 而非 `test_gt + qi * test_gt_d`。
- 修复: 将 `gt[j * gtd]` 改为 `gt[j]`，调用处改为 `test_gt + qi * test_gt_d`。

**问题 2: MPI_Bcast 报错 `MPI_ERR_TRUNCATE: message truncated`（np≥2 时）**
- 原因: `MPI_Bcast` 是集体通信操作，所有 rank 必须同时调用。代码中质心广播被 `if(rank == 0)` 包裹，导致非 rank 0 进程跳过了该调用。当 rank 0 发送质心数据时，rank 1 却在等待接收 inverted list 数据，消息类型/大小不匹配。
- 修复: 移除 `if(rank == 0)` 条件，让所有 rank 都调用 `MPI_Bcast`。

**问题 3: reduce 模式 recall 异常偏高 (3.70, 恰好是 np=4 时的 4×0.925)**
- 原因: reduce 模式使用 `MPI_Allgatherv`，所有 rank 都有完整的搜索结果并各自计算 recall。最终 `MPI_Reduce(MPI_SUM)` 将所有 rank 的 recall 求和，导致结果放大了 np 倍。
- 修复: 在 reduce 模式中，添加 `if(rank == 0)` 条件，仅 rank 0 计算 recall。

### 基准测试结果

**环境**: WSL2, x86_64, AVX2, IVF nlist=256, nprobe=8, 2000 queries, k=10

**串行基线** (main_ivf_avx2, serial mode):

| 指标 | 值 |
|------|-----|
| Recall | 0.92465 |
| Latency (us) | 196.3 |

**MPI IVF gather 模式**:

| np | Recall | Latency (us) | 加速比 | 效率 |
|----|--------|-------------|--------|------|
| 1 | 0.92465 | 184.9 | 1.00× | 100% |
| 2 | 0.92465 | 105.8 | 1.75× | 87.5% |
| 4 | 0.92465 | 66.7 | 2.77× | 69.3% |
| 8 | 0.92465 | 47.7 | 3.87× | 48.4% |

**MPI IVF reduce 模式** (np=4):

| np | Recall | Latency (us) |
|----|--------|-------------|
| 4 | 0.92465 | 127.6 |

**分析**:
- Recall 在所有进程数下完全一致 (0.92465)，验证了 MPI 并行化不影响搜索正确性
- 加速比随进程数增加但非线性的原因: ① 通信开销（broadcast/gather）② 负载不均衡（不同簇包含的向量数量差异大，块分配无法保证均匀）③ rank 0 集中 merge 开销
- Reduce 模式（Allgatherv）比 gather 模式更慢，因为 Allgatherv 让所有 rank 交换完整数据，通信量更大。对于小 k(=10)，gather 到单进程 merge 更高效

### 待优化项

- 负载均衡：当前按簇编号块分配，未考虑簇大小差异。可改为按向量数均匀分配
- 通信：对每 query 单独做 MPI_Gather 开销大，可考虑批量处理
- 非阻塞通信：使用 MPI_Isend/MPI_Irecv 隐藏通信延迟

## 阶段 3：MPI HNSW 搜索（方案 B：数据划分）

### 2026-06-03 实现

**新建文件:**
- `mpi/main_hnsw_mpi.cc` — HNSW MPI 主程序（方案 B + 方案 A 的 K-Means）

**设计方案:**
- 方案 B：数据集划分 → 各 rank 独立构建 HNSW → MPI 并行搜索 → top-k merge
- 两种划分方式: random（按索引范围块划分）和 kmeans（K-Means 聚类划分）
- 标签管理：各 rank 使用原始全局索引作为 HNSW label，确保 merge 时索引一致性

### 遇到的问题

**问题 6: hnsw_wrapper.h include 路径不兼容**
- 症状: `fatal error: hnswlib.h: No such file or directory`
- 原因: `hnsw_wrapper.h` 使用 `#include "hnswlib.h"`（相对路径），但实际文件在 `hnswlib/hnswlib/hnswlib.h`。父目录编译时也可能存在问题（经验证 main_hnsw.cc 同样编译失败）
- 修复: 在 MPI 代码中直接使用 `#include "../hnswlib/hnswlib/hnswlib.h"`，绕过有问题的 wrapper

**问题 7: HNSW recall 在 np≥2 时接近 0**
- 症状: np=1 时 recall=0.973 正常，np=2 时 recall=0.00005
- 原因: `merge_topk()` 使用 `f < result.top().first` 保留较小的距离值（IVF语义：越小=越好）。但 HNSW 代码中将距离转换为内积值（越大=越好），语义相反，导致保留的是最差的 k 个结果
- 修复: 不再转换为内积值，直接使用 hnswlib 原始距离（1.0-inner_product，越小=越好），与 IVF 语义统一

**问题 8: K-Means 划分不提升召回率**
- 原因分析: DEEP100K 数据本身没有明显的聚类结构，K-Means 划分与随机划分效果相近，但额外增加了聚类计算开销（2-10× 慢）
- 结论: 对于此数据集，random 划分（按索引范围）既简单又高效

**Random 划分:**

**K-Means 划分:**

**关键发现: 召回率随进程数增加而提升**
- 原因: 将数据分区后，每个 HNSW 的搜索空间缩小，图搜索在更小的局部空间中更精确
- 合并来自多个分区的独立搜索结果，能覆盖更多真正的近邻
- 这是分区近似搜索中的已知优势现象（ensemble effect）
- 此结果对报告中的 recall-latency trade-off 分析非常有价值

### 2026-06-03 综合分析

**效率下降原因**:

2. Amdahl 定律：串行部分（rank 0 构建索引、质心扫描、merge）占比随 np 增大
3. 负载不均衡：部分 rank 获得更多向量，成为瓶颈

### 六、通信-计算分析

**IVF 通信模式**:
- 初始化: MPI_Bcast(base 37MB) + MPI_Bcast(lists ~400KB) → 一次性开销
- 搜索: 每 query 一轮 MPI_Bcast(query 384B) + MPI_Gather(结果 ~80B/rank)
- 通信/计算比: ~0.1%（计算主导，适合 MPI）

**HNSW 通信模式**:
- 初始化: MPI_Bcast(base 37MB) → 一次性开销
- 搜索: 每 query 一轮 MPI_Gather(结果 ~80B/rank)
- 通信/计算比: ~0.05%（更低，HNSW 搜索计算量小）

### 七、混合并行负优化分析

| 模式 | total_parallel=8 | Latency (us) | vs 纯 MPI np=8 |
|------|-----------------|-------------|-----------------|
| 纯 MPI | np=8 | 51.2 | 1.00× |
| mpi_omp_query | np=4,nt=2 | 333.7 | 6.5× 慢 |
| mpi_omp_cluster | np=4,nt=2 | 904 | 17.7× 慢 |

**原因**:
1. OpenMP 线程管理开销约 10-50us，与 query 延迟同量级
2. mpi_omp_cluster: 每 query 一轮 MPI_Gather（2000轮），per-query 通信延迟累积
3. 纯 MPI 块分配天然减少每 rank 计算量
4. 混合并行在更大数据/更多查询的强扩展场景中可能更有效

### 分析结论

1. **IVF 纯 MPI 按簇分配** 是最均衡的方案（recall 0.925, 51us @ np=8, 3.9×加速比）
2. **HNSW 数据划分** 最精确（recall 0.998 @ np=8, 且 recall 随 np 增长）
3. **混合并行在本实验规模下不适用**，纯 MPI 已足够有效
4. **通信开销不是瓶颈**（通信/计算比 < 0.1%）

## 阶段 2：MPI IVF + OpenMP 混合并行

### 2026-06-03 实现

**新建文件:**
- `mpi/mpi_ivf_hybrid.h` — MPI+OpenMP 混合并行 IVF 搜索

**修改文件:**
- `mpi/main_ivf_mpi.cc` — 添加 `IVF_MODE` 分发逻辑，支持 `mpi_omp_query` 和 `mpi_omp_cluster`

**新增环境变量:**
- `IVF_MODE=mpi_omp_query | mpi_omp_cluster | mpi_gather` (默认)
- `NUM_THREADS` — 每个 MPI rank 内 OpenMP 线程数

**两种混合模式:**

1. **mpi_omp_query**: MPI 分散查询（Scatterv）+ rank 内 OpenMP 并行处理查询
   - 每个 rank 做完整质心扫描 + 全部 nprobe 个簇扫描
   - 通信: 一次 MPI_Scatterv 分发查询 + 一次 MPI_Gatherv 收集结果
   
2. **mpi_omp_cluster**: MPI 广播查询 + 按簇分配 + rank 内 OpenMP 扫描簇
   - 每个 rank 只扫描 1/nranks 的簇，但需每个 query 做 MPI_Gather
   - 通信: 2000 轮 MPI_Gather（每个 query 一轮）

### 遇到的问题

**问题 4: mpi_omp_cluster 模式死锁**
- 症状: 程序挂起无输出，所有 MPI rank 卡在等待状态
- 原因: `MPI_Reduce` 是集体通信操作，所有 rank 必须同时调用。代码中第二个 `MPI_Reduce` 被 `if(rank == 0)` 包裹，导致只有 rank 0 调用，其他 rank 不调用，形成死锁
- 修复: 将两个 `MPI_Reduce` 调用都移到 `if(rank == 0)` 外面，所有 rank 共同参与

**问题 5: mpi_omp_query 比纯 MPI 慢很多（负优化）**
- 原因分析:
  1. mpi_omp_query 中每个 rank 拥有完整 inverted list（不过滤簇），每个查询需扫描所有选中的簇，与串行工作量相同
  2. OpenMP 线程管理有额外开销（创建/销毁线程池）
  3. 纯 MPI 模式中，查询被广播到所有 rank，每个 rank 仅扫描 1/nranks 的簇，通信是批量广播而非 per-query
- 结论: 对于本实验的查询数(2000)和数据集大小(100k)，纯 MPI 按簇分配是最优策略

### 基准测试结果

**纯 MPI gather (基线):**

| np | Latency (us) | Speedup |
|----|-------------|---------|
| 1 | 216.8 | 1.00× |
| 2 | 119.1 | 1.82× |
| 4 | 75.5 | 2.87× |
| 8 | 51.8 | 4.19× |

**mpi_omp_query (总并行度=8):**

| np | nt | Latency (us) | vs 纯 MPI np=8 |
|----|-----|-------------|---------------|
| 2 | 4 | 475.2 | 9.2× 慢 |
| 4 | 2 | 333.7 | 6.4× 慢 |
| 8 | 1 | 377.0 | 7.3× 慢 |

**mpi_omp_cluster (总并行度=8):**

| np | nt | Latency (us) | vs 纯 MPI np=8 |
|----|-----|-------------|---------------|
| 2 | 4 | 433 | 8.4× 慢 |
| 4 | 2 | 904 | 17.4× 慢 |
| 8 | 1 | 603 | 11.6× 慢 |

**负优化分析 (供报告使用):**
- 混合并行在本实验规模下不如纯 MPI 的原因:
  1. OpenMP 线程管理开销 ≈ 纯 MPI 的 cluster 分配已足够均衡
  2. mpi_omp_query: 每 rank 有全量索引，每个查询做全量扫描（与串行一致），并行只体现在查询级多 rank 同时处理
  3. mpi_omp_cluster: 每 query 一轮 MPI_Gather (2000 轮)，通信次数过多
  4. 纯 MPI 按簇分配天然减少每 rank 计算量 (1/np)，且通信是批量广播
- 混合并行在更大规模数据或更多查询时可能更有优势，值得在报告中讨论

## 阶段 6：补充探索 — IVF 划分策略对比

### 实现
修改 `mpi/main_ivf_mpi.cc`，添加 `IVF_PARTITION=balanced` 模式：
- **block**: 按簇编号连续块分配（原有）
- **balanced**: 贪心算法按向量总数均匀分配

**贪心算法**: 将簇按大小降序排列，每次将最大簇分配给当前总向量数最少的 rank。

### 负载均衡对比

| np | block 不均衡度 | balanced 不均衡度 |
|----|---------------|-------------------|
| 2 | 6.3% | 0.01% |
| 4 | 16.3% | 0.02% |
| 8 | 62.3% | 0.05% |

### latency 对比

| np | block (us) | balanced (us) | 差异 |
|----|-----------|---------------|------|
| 2 | 142.9 | 161.5 | +13% (更差) |
| 4 | 90.2 | 90.1 | 持平 |
| 8 | 67.8 | 61.3 | -9.6% (更好) |

**分析:**
- Balanced 在低 np 时反而更慢：贪心分配破坏了块分配的 cache 友好性（相邻簇对应相近的质心空间区域，连续扫描对 cache 更友好），负载均衡收益不足以抵消 cache locality 损失
- Balanced 在高 np 时更好：np=8 时 block 不均衡度高达 62%，balanced 降低到 0.05%，收益超过 cache 损失
- **结论**: 存在负载均衡与 cache locality 的 trade-off。低 np 选 block，高 np 选 balanced

## 阶段 7：补充探索 — 动态任务池（主从工作窃取）

### 实现
在 `mpi/main_ivf_mpi.cc` 添加 `IVF_MODE=mpi_dynamic`：
- rank 0 = master，维护查询计数器
- worker (rank 1..P-1) 空闲时主动向 master 请求下一个查询
- master 收到请求后分配查询索引，worker 完成后发回结果
- 实现真正的动态负载均衡——快的 rank 处理更多查询

### block 分配下 static vs dynamic

| np | static (us) | dynamic (us) | 改善 |
|----|------------|-------------|------|
| 2 | 147.5 | 125.9 | -14.6% |
| 4 | 93.8 | 82.5 | -12.1% |
| 8 | 58.4 | 56.1 | -4.0% |

### balanced 分配下 static vs dynamic

| np | static (us) | dynamic (us) | 改善 |
|----|------------|-------------|------|
| 2 | 139.7 | 116.7 | -16.5% |
| 4 | 77.9 | 83.3 | +6.9% (更差) |
| 8 | 61.5 | 61.5 | 持平 |

### 关键发现

1. **Dynamic 对 block 分配始终有效**：-4~15%，因为 block 的不均衡使某些 rank 更快完成固定分配，dynamic 让空闲 rank 继续工作
2. **Dynamic 对 balanced 分配反效或持平**：负载已均衡时，per-query MPI 通信（3 条消息/query）成为额外开销
3. **np=8 block 的改善仅 4%**：虽然不均衡度 62%，但每 query 只搜索 nprobe 个选中的簇（而非所有分配的簇），实际计算时间受选中簇的总向量数影响，与分配的总向量数不完全正比
4. **Dynamic + balanced @ np=4 反效**（83.3 vs 77.9）：负载已极均衡时，dynamic 的通信开销（2000×3 条额外 MPI 消息）成为纯负担

### 总结：三种负载均衡策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| block | cache 友好，通信简单 | 高 np 时不均衡严重 | 低 np（≤4） |
| balanced | 负载极均衡 | 破坏 cache locality | 高 np（≥8） |
| dynamic | 自适应，补偿不均衡 | per-query 通信开销 | block+中等 np 组合 |
