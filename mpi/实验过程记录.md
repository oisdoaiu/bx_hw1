## 阶段 1：MPI IVF 基础实现

### 2026-06-03 初始化

**创建文件:**
- `mpi/mpi_utils.h` — MPI 通用工具（序列化、top-k merge、recall 计算）
- `mpi/main_ivf_mpi.cc` — IVF MPI 主程序

**设计决策:**

1. **按簇分配 inverted list**（而非按向量 ID 切分）
   - 理由: 保持每个 rank 上 SIMD 扫描的连续性，每个 rank 获得若干完整的簇
   - 按向量 ID 切分会破坏 cache locality，且需要在 rank 间传递大量 base 向量数据

2. **使用全局变量传递 IVF 索引**
   - `ivf_simd.h` 使用 `static` 全局变量 (`g_ivf_centroids`, `g_ivf_lists`, `g_ivf_nlist`, `g_ivf_base`)
   - 每个 MPI rank 设置自己的全局变量副本，避免跨进程共享
   - 优点: 不需要修改已有的 `ivf_search()` 函数签名

3. **top-k 归并方式**
   - 提供两种模式对比:
     - `gather`: MPI_Gather 收集所有 rank 的 local top-k 到 rank 0，串行 merge
     - `reduce`: 自定义 MPI_Op 实现树形规约（O(log P) 轮通信）
   - 使用环境变量 `IVF_MERGE_MODE` 切换

4. **时序测量**
   - 使用 `MPI_Wtime()` 替代 `gettimeofday()`，因为 MPI 提供了跨进程的时间基准
   - 在 rank 0 上测量端到端延迟（包含通信开销）

### 遇到的问题和解决方法

**问题 1: recall 接近 0 (0.00065)**
- 原因: `compute_recall()` 中 ground truth 索引错误。函数签名接受指向当前 query 的 gt 起始位置的指针，但内部使用了 `gt[j * gtd]`（应为 `gt[j]`，因为指针已指向正确位置）。同时调用处使用了 `test_gt + qi` 而非 `test_gt + qi * test_gt_d`。
- 修复: 将 `gt[j * gtd]` 改为 `gt[j]`，调用处改为 `test_gt + qi * test_gt_d`。

**问题 2: MPI_Bcast 报错 `MPI_ERR_TRUNCATE: message truncated`（np≥2 时）**
- 原因: `MPI_Bcast` 是集体通信操作，所有 rank 必须同时调用。代码中质心广播被 `if(rank == 0)` 包裹，导致非 rank 0 进程跳过了该调用。当 rank 0 发送质心数据时，rank 1 却在等待接收 inverted list 数据，消息类型/大小不匹配。
- 修复: 移除 `if(rank == 0)` 条件，让所有 rank 都调用 `MPI_Bcast`。

**问题 3: reduce 模式 recall 异常偏高 (3.70, 恰好是 np=4 时的 4×0.925)**
- 原因: reduce 模式使用 `MPI_Allgatherv`，所有 rank 都有完整的搜索结果并各自计算 recall。最终 `MPI_Reduce(MPI_SUM)` 将所有 rank 的 recall 求和，导致结果放大了 np 倍。
- 修复: 在 reduce 模式中，添加 `if(rank == 0)` 条件，仅 rank 0 计算 recall。

### 基准测试结果

**环境**: WSL2, x86_64, AVX2, IVF nlist=256, nprobe=8, 2000 queries, k=10

**串行基线** (main_ivf_avx2, serial mode):

| 指标 | 值 |
|------|-----|
| Recall | 0.92465 |
| Latency (us) | 196.3 |

**MPI IVF gather 模式**:

| np | Recall | Latency (us) | 加速比 | 效率 |
|----|--------|-------------|--------|------|
| 1 | 0.92465 | 184.9 | 1.00× | 100% |
| 2 | 0.92465 | 105.8 | 1.75× | 87.5% |
| 4 | 0.92465 | 66.7 | 2.77× | 69.3% |
| 8 | 0.92465 | 47.7 | 3.87× | 48.4% |

**MPI IVF reduce 模式** (np=4):

| np | Recall | Latency (us) |
|----|--------|-------------|
| 4 | 0.92465 | 127.6 |

**分析**:
- Recall 在所有进程数下完全一致 (0.92465)，验证了 MPI 并行化不影响搜索正确性
- 加速比随进程数增加但非线性的原因: ① 通信开销（broadcast/gather）② 负载不均衡（不同簇包含的向量数量差异大，块分配无法保证均匀）③ rank 0 集中 merge 开销
- Reduce 模式（Allgatherv）比 gather 模式更慢，因为 Allgatherv 让所有 rank 交换完整数据，通信量更大。对于小 k(=10)，gather 到单进程 merge 更高效

### 待优化项

- 负载均衡：当前按簇编号块分配，未考虑簇大小差异。可改为按向量数均匀分配
- 通信：对每 query 单独做 MPI_Gather 开销大，可考虑批量处理
- 非阻塞通信：使用 MPI_Isend/MPI_Irecv 隐藏通信延迟

## 阶段 2：MPI IVF + OpenMP 混合并行

### 2026-06-03 实现

**新建文件:**
- `mpi/mpi_ivf_hybrid.h` — MPI+OpenMP 混合并行 IVF 搜索

**修改文件:**
- `mpi/main_ivf_mpi.cc` — 添加 `IVF_MODE` 分发逻辑，支持 `mpi_omp_query` 和 `mpi_omp_cluster`

**新增环境变量:**
- `IVF_MODE=mpi_omp_query | mpi_omp_cluster | mpi_gather` (默认)
- `NUM_THREADS` — 每个 MPI rank 内 OpenMP 线程数

**两种混合模式:**

1. **mpi_omp_query**: MPI 分散查询（Scatterv）+ rank 内 OpenMP 并行处理查询
   - 每个 rank 做完整质心扫描 + 全部 nprobe 个簇扫描
   - 通信: 一次 MPI_Scatterv 分发查询 + 一次 MPI_Gatherv 收集结果
   
2. **mpi_omp_cluster**: MPI 广播查询 + 按簇分配 + rank 内 OpenMP 扫描簇
   - 每个 rank 只扫描 1/nranks 的簇，但需每个 query 做 MPI_Gather
   - 通信: 2000 轮 MPI_Gather（每个 query 一轮）

### 遇到的问题

**问题 4: mpi_omp_cluster 模式死锁**
- 症状: 程序挂起无输出，所有 MPI rank 卡在等待状态
- 原因: `MPI_Reduce` 是集体通信操作，所有 rank 必须同时调用。代码中第二个 `MPI_Reduce` 被 `if(rank == 0)` 包裹，导致只有 rank 0 调用，其他 rank 不调用，形成死锁
- 修复: 将两个 `MPI_Reduce` 调用都移到 `if(rank == 0)` 外面，所有 rank 共同参与

**问题 5: mpi_omp_query 比纯 MPI 慢很多（负优化）**
- 原因分析:
  1. mpi_omp_query 中每个 rank 拥有完整 inverted list（不过滤簇），每个查询需扫描所有选中的簇，与串行工作量相同
  2. OpenMP 线程管理有额外开销（创建/销毁线程池）
  3. 纯 MPI 模式中，查询被广播到所有 rank，每个 rank 仅扫描 1/nranks 的簇，通信是批量广播而非 per-query
- 结论: 对于本实验的查询数(2000)和数据集大小(100k)，纯 MPI 按簇分配是最优策略

**负优化分析 (供报告使用):**
- 混合并行在本实验规模下不如纯 MPI 的原因:
  1. OpenMP 线程管理开销 ≈ 纯 MPI 的 cluster 分配已足够均衡
  2. mpi_omp_query: 每 rank 有全量索引，每个查询做全量扫描（与串行一致），并行只体现在查询级多 rank 同时处理
  3. mpi_omp_cluster: 每 query 一轮 MPI_Gather (2000 轮)，通信次数过多
  4. 纯 MPI 按簇分配天然减少每 rank 计算量 (1/np)，且通信是批量广播
- 混合并行在更大规模数据或更多查询时可能更有优势，值得在报告中讨论
